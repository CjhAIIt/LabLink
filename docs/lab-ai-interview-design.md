# 实验室管理系统 AI 面试功能设计文档

## 1. 文档目的

本文用于指导“实验室管理系统”新增 AI 面试功能，并说明如何参考当前仓库 `ai-interview-standalone` 的实现方式进行复用或模仿。

当前仓库更适合作为一个“交互原型”和“AI 面试链路样例”，不建议原样搬运到正式系统。正式系统需要补齐权限、次数控制、数据落库、管理员查看、异常处理等业务能力。

## 2. 参考旧项目可复用能力

当前项目已经具备以下可直接参考的能力：

- 多步骤面试流程：角色选择、语音设置、虚拟形象、正式问答、报告生成。
- 多轮 AI 追问：根据学生回答继续提问，而不是固定问卷。
- 报告生成：面试结束后生成 `score + tags + summary` 的结构化结果。
- 语音输入：浏览器端通过 Web Speech API 进行语音识别。
- 语音播报：通过 TTS 将 AI 面试官问题朗读出来。
- 虚拟面试官展示：通过 Live2D 增强模拟面试体验。

## 3. 新功能定位

实验室管理系统中的 AI 面试应拆分为两类场景：

### 3.1 模拟面试

- 面向学生自主练习。
- 学生根据自己学习过的模块选择面试方向。
- 主要目标是让学生熟悉真实面试节奏，提前暴露知识薄弱点。
- 模拟结果只在当前界面展示，不保存到数据库。
- 建议默认不限次数。

### 3.2 实验室正式 AI 面试

- 面向实验室招新或实验室内部考核。
- 与线下面试是两套独立流程，不能互相替代。
- 每个学生仅有 2 次正式 AI 面试机会。
- 正式 AI 面试结果需要保存，供管理员查看。
- AI 评分和 AI 评价仅作为参考依据之一，不能直接替代人工最终决策。

## 4. 功能目标

本功能需要同时满足以下目标：

- 学生可以低成本进行模块化模拟面试。
- 管理员可以通过正式 AI 面试快速获得第一轮参考评分。
- 系统要明确区分“练习”和“正式考核”，避免数据混淆。
- 正式考核要有次数限制和结果留痕。
- 面试结果要让学生感受到真实面试的提问节奏和压力。

## 5. 用户角色

### 5.1 学生

- 进入模拟面试页面，自主选择模块并开始练习。
- 进入正式 AI 面试页面，查看剩余次数并开始正式面试。
- 查看本次面试结束后的 AI 评分、标签、评价。

### 5.2 管理员

- 配置可面试模块。
- 配置正式 AI 面试是否开放。
- 查看学生正式 AI 面试记录。
- 查看 AI 评分、AI 评价、面试时间、剩余次数消耗情况。
- 在特殊情况下手动重置或补偿次数。

## 6. 业务规则

### 6.1 模拟面试规则

- 学生选择模块后可直接开始。
- 模拟问答过程不写入数据库。
- 模拟报告只用于当前页面展示。
- 页面刷新或退出后，模拟数据可直接丢弃。
- 若后续有教学分析需求，可考虑只记录匿名统计，不记录个人详细内容。

### 6.2 正式 AI 面试规则

- 每名学生最多 2 次机会。
- 机会次数建议在“正式开始并成功拿到第一题”后扣减 1 次。
- 正式 AI 面试结果需要保存到数据库。
- 正式 AI 面试应记录开始时间、结束时间、模块、得分、评价、使用次数、会话状态。
- 正式 AI 面试记录默认不可由学生删除或修改。
- 管理员可以查看正式记录，但不应修改 AI 原始评分结果。

### 6.3 结果使用规则

- AI 评分只能作为管理员筛选时的辅助参考。
- 管理员仍需结合线下面试、项目经历、学习态度进行综合判断。
- 学生端应明确提示：“AI 面试结果仅供参考，不代表最终录取结果。”

## 7. 功能需求

### 7.1 学生端功能

- 入口一：模拟面试。
- 入口二：正式 AI 面试。
- 模块选择。
- 面试说明展示。
- 剩余次数展示，仅正式 AI 面试显示。
- 面试过程中的文字问答。
- 可选语音输入。
- 可选 AI 播报。
- 面试结束后展示评分报告。

### 7.2 管理端功能

- 模块管理。
- 正式 AI 面试开关控制。
- 每名学生正式面试次数查看。
- 正式 AI 面试记录查询。
- 按模块、得分区间、时间、学生姓名筛选。
- 查看单次面试的评分、评价、标签、摘要。
- 异常场景下补偿次数或作废记录。

### 7.3 AI 能力要求

- 根据所选模块动态生成问题。
- 每次只问一个问题。
- 能根据学生回答进行追问。
- 在多轮对话结束后给出结构化评分结果。
- 输出统一格式，便于系统展示和管理端筛选。

## 8. 建议流程

### 8.1 模拟面试流程

1. 学生进入模拟面试页面。
2. 选择学习模块。
3. 阅读说明后开始面试。
4. AI 按模块连续提问。
5. 学生通过文本或语音回答。
6. 面试结束后生成即时评分和评价。
7. 报告仅展示，不入库。

### 8.2 正式 AI 面试流程

1. 学生进入正式 AI 面试页面。
2. 系统展示剩余次数。
3. 学生选择本次面试模块。
4. 学生确认开始后，系统创建正式面试会话。
5. 成功生成首题后扣减 1 次机会。
6. 面试结束后保存评分、评价、时间、模块等信息。
7. 管理员在后台查看正式 AI 面试结果。

## 9. 模块设计建议

建议把“角色选择”改造成“模块选择”，避免沿用旧项目中过于偏前端演示化的角色定义。

可选模块示例：

- Java 基础
- Spring / Spring Boot
- 前端基础
- Vue 或 React
- MySQL
- 数据结构与算法
- 计算机网络
- 操作系统
- 项目经历表达

模块来源建议支持两种方式：

- 管理员后台维护模块列表。
- 管理员为每个模块维护难度、关键词、评分侧重点。

## 10. 数据设计建议

模拟面试不入库，因此数据库设计主要服务正式 AI 面试。

### 10.1 `ai_interview_module`

用于维护面试模块。

建议字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `module_name` | 模块名称 |
| `module_code` | 模块编码 |
| `description` | 模块说明 |
| `prompt_template` | 该模块的提示词模板 |
| `score_rule` | 评分规则配置 |
| `status` | 是否启用 |
| `create_time` | 创建时间 |

### 10.2 `ai_interview_record`

用于保存正式 AI 面试结果。

建议字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `student_id` | 学生 ID |
| `student_name` | 学生姓名，冗余可选 |
| `module_id` | 面试模块 ID |
| `attempt_no` | 第几次正式 AI 面试 |
| `score` | AI 评分 |
| `tags_json` | 标签列表 |
| `summary` | AI 评价摘要 |
| `conversation_json` | 对话记录，可选 |
| `status` | 已完成、异常中断、作废 |
| `start_time` | 开始时间 |
| `end_time` | 结束时间 |
| `create_time` | 创建时间 |

### 10.3 次数控制方式

有两种实现方式，推荐第二种：

- 方式一：单独建立次数表，专门记录剩余次数。
- 方式二：通过正式 AI 面试记录表统计已使用次数。

推荐方式二，原因如下：

- 数据更统一。
- 不容易出现“记录表和次数表不一致”。
- 后续便于审计。

若采用方式二，则剩余次数计算公式为：

`剩余次数 = 2 - 已完成或已扣减次数`

## 11. 接口设计建议

如果未来仍然采用前后端分离结构，建议保留旧项目中“对话接口 + 报告接口”的思路，但接口语义要更贴近业务。

### 11.1 学生端接口

- `GET /api/ai-interview/modules`
  - 获取模块列表。
- `GET /api/ai-interview/formal/chances`
  - 获取当前学生正式 AI 面试剩余次数。
- `POST /api/ai-interview/session/start`
  - 开始一次模拟或正式面试。
- `POST /api/ai-interview/chat`
  - 获取 AI 下一轮问题。
- `POST /api/ai-interview/report`
  - 生成本次面试报告。
- `POST /api/ai-interview/session/finish`
  - 正式面试结束时保存结果。

### 11.2 管理端接口

- `GET /api/admin/ai-interview/records`
  - 分页查询正式面试记录。
- `GET /api/admin/ai-interview/records/{id}`
  - 查看单次正式面试详情。
- `POST /api/admin/ai-interview/modules`
  - 新增模块。
- `PUT /api/admin/ai-interview/modules/{id}`
  - 修改模块。
- `POST /api/admin/ai-interview/records/{id}/invalidate`
  - 作废一次异常记录。
- `POST /api/admin/ai-interview/students/{studentId}/reset-chance`
  - 特殊情况下补偿次数。

## 12. Prompt 与评分建议

正式系统不要把提示词只写死在代码里，建议按模块配置。

### 12.1 对话 Prompt 建议

对话 Prompt 至少包含以下信息：

- 当前是模拟面试还是正式 AI 面试。
- 当前模块名称。
- 面试目标人群。
- 提问风格。
- 每次只提一个问题。
- 根据学生回答进行追问。
- 追问轮数上限。
- 在结束时输出结束语，不直接输出评分。

### 12.2 报告 Prompt 建议

报告 Prompt 建议要求模型统一输出：

- `score`
- `tags`
- `summary`
- `strengths`
- `weaknesses`
- `suggestions`

如果你想先保持和旧项目一致，也可以第一阶段只输出：

- `score`
- `tags`
- `summary`

### 12.3 评分维度建议

可以按模块统一采用以下维度：

- 基础知识掌握。
- 理解深度。
- 项目实践能力。
- 问题分析能力。
- 表达清晰度。

建议总分 100 分，管理员端展示时同时显示：

- 总分。
- 维度概览。
- 总评摘要。

## 13. 与旧项目的映射关系

为了让 Claude 或其他代码助手更容易参考本仓库实现，建议按下面的映射关系迁移。

### 13.1 可直接复用的设计思路

- 旧项目的多步骤页面流程可以复用，但页面文案要改成实验室场景。
- 旧项目的 `chatHistory` 会话管理方式可以复用。
- 旧项目的 `chat` 与 `report` 双接口模式可以复用。
- 旧项目的“每次只提一个问题”的问答节奏可以复用。
- 旧项目的评分报告展示方式可以复用。

### 13.2 需要改造的部分

- 旧项目中的角色选择要改成模块选择。
- 旧项目默认是独立应用，新系统需要接入实验室管理系统登录态。
- 旧项目没有次数限制，新系统要增加正式 AI 面试限次逻辑。
- 旧项目没有管理员查看页，新系统要增加后台记录页。
- 旧项目模拟与正式场景未分离，新系统要拆成两套业务入口。

### 13.3 不建议直接照搬的部分

- Live2D 形象是加分项，不是核心业务，可后置。
- TTS 可保留为可选功能，不必作为第一阶段必做项。
- 旧项目的服务端 STT 方案未完成，不应直接用于正式环境。

## 15. 开发优先级建议

### 第一阶段

- 完成模块选择。
- 完成模拟面试。
- 完成正式 AI 面试 2 次限制。
- 完成正式结果入库。
- 完成管理员记录查看。

### 第二阶段

- 增加维度化评分。
- 增加管理员筛选和统计。
- 增加 TTS 播报。
- 增加对话记录回放。

### 第三阶段

- 增加 Live2D 或虚拟面试官形象。
- 增加更细粒度的 Prompt 配置。
- 增加题目难度分层和模块题库策略。

## 16. 验收标准

- 学生可以在模拟面试中自由选择模块进行练习。
- 模拟面试结果不写入数据库。
- 正式 AI 面试和模拟面试入口明确分离。
- 正式 AI 面试每个学生最多只能完成 2 次。
- 正式 AI 面试结果可以被管理员查询和查看。
- 管理员能看到 AI 评分和 AI 评价。
- 页面明确提示 AI 结果仅供参考，不替代人工最终判断。

## 17. 结论

这个功能最关键的不是“把旧项目搬进来”，而是把旧项目中的 AI 面试交互能力，转成实验室管理系统里的正式业务能力。

最适合当前需求的落地方式是：

- 复用旧项目的问答链路和评分链路。
- 把角色选择改造成模块选择。
- 把单机演示流程改造成“模拟面试 + 正式 AI 面试”双模式。
- 用正式记录表完成 2 次机会控制和管理员查看。
