# 安徽信息工程学院全校版改造说明

本文档对应用户提供的《安徽信息工程学院实验室管理平台（全校版）Codex 需求文档》，用于说明当前代码库已完成的对齐项、现存底座能力与后续待办。

## 需求文档核心要求

需求文档强调以下改造目标：

- 从单实验室 / 单学院语境升级为全校版平台
- 按“学校 - 学院 - 实验室 - 成员”组织业务
- 注册时区分学生与老师
- 老师注册需审批后激活
- 老师可申请创建实验室
- 老师可申请成为实验室指导老师，且同一时间每个实验室仅 1 位指导老师
- 建立学院管理员、学校管理员双层治理能力
- 增加审批日志、状态机、历史记录和权限控制

## 当前代码库已具备的底座

后端已经具备以下基础能力：

- 学院主数据表 `t_college`
- 实验室主表 `t_lab`
- 用户表 `t_user`
- 实验室创建申请表 `t_lab_create_apply`
- 学院审核 / 学校审核字段
- 平台岗位表 `t_platform_post`
- 用户身份表 `t_user_identity`
- 实验室教师关系表 `t_lab_teacher_relation`
- 学院管理员 / 学校管理员兼容访问模型

这些内容说明，当前项目并不是从零开始，已经具备承接“全校版”改造的数据库和权限基础。

## 本次已落地修改

本次改动优先完成了“学校版语义统一”和“初始化主数据切换”：

- 平台名称统一为“安徽信息工程学院实验室管理平台”
- 浏览器标题、登录页、注册页、平台介绍页、管理端 / 学生端侧边栏完成学校版改造
- 注册页增加“所属学院”选择，匹配全校版主数据要求
- 新增 `/intro` 平台介绍页，展示学校版角色和学院范围
- 后端邮件发送名称改为学校版平台名称
- 初始化 SQL 改为学校版默认学院、实验室、公告和示例账号
- `DatabaseSchemaUpdate` 增加学校版学院主数据的 upsert 逻辑

## 与需求文档仍有差距的部分

以下功能文档中明确要求，但当前仓库还没有完整闭环：

- 老师注册申请与审批激活
- 老师注册资料扩展字段，如工号、职称、证明材料
- 指导老师申请流程
- 现任指导老师同意 / 拒绝节点
- 指导老师历史变更记录
- 统一申请单主表与审批日志表
- 后台老师审核页、指导老师审核页

## 推荐下一阶段开发顺序

建议按照下面顺序继续推进：

1. 重构注册模型，区分学生注册与老师注册申请
2. 新增老师注册申请表和审批接口
3. 打通老师登录后“待审核状态页”
4. 新增指导老师申请 / 变更申请表
5. 新增审批日志和状态机
6. 补齐后台审批页面和测试用例

## 当前结论

当前仓库已经从“旧竞赛版 / 单学院语境”切换到了“安徽信息工程学院全校版”的品牌和主数据语境，但还没有完整实现需求文档中的老师审批链与指导老师流转闭环。

如果继续开发，优先级最高的是：

- 老师注册审批链
- 指导老师申请与变更
- 审批日志 / 历史记录
